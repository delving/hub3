    Introduction

    Welcome to the V2 Search API! You can use our API to access V2 Search API endpoints, which can get information on all the RDF records stored in our database

    We have language bindings in Shell! In the future, we intend to extend that with Golang, Python, and Javascript. You can view code examples in the dark area to the right, and you can switch the programming language of the examples with the tabs in the top right.

    The all API request and responses are modelled using Google Protocol buffers. Our language bindings are generated from these shared proto files. When consuming the API we recommend that you generate models to query and to read the response from the api.proto file available in the Rapid repository available at github.

    This example API documentation page was created with Slate.

    Authentication

    To authorize, use this code:

    # With shell, you can just pass the correct header with each request
curl "api_endpoint_here"
  -H "Authorization: {apiKEY}"

    Make sure to replace {apiKEY} with your API key.

    V2 Search API uses API keys to allow full access to the API. You can register a new v2 API key at our developer portal. Anonymous access to the read-only APIs might be throttled.

    The V2 Search API expects for the API key to be included in all private API requests to the server in a header that looks like the following:

    Authorization: {apiKEY}

    You must replace {apiKEY} with your personal API key.

    RDF Search

    In the next sections, these high level structures will be described.

    Pager

    The 'pager' block contains all the information to page through the search result. The scrollID contains a protobuf serialised version of the original search request. This is used for paging. The scrollID is empty when the end of the search results have been reached. It also gives back the total number of results and the current cursor of the search results.

    The pager can be used for creating 'inifite scroll' functionality or to update the results in an HTML page through an AJAX call. Underwater the pager contains functionality to offer efficient deep paging without keeping state in the client or on the server.

    For output formats where no pager block can be added - e.g. geobuf, geojson - the pager keys are added to the response headers as P_SCROLL_ID, P_CURSOR, P_TOTAL.

    Once you start paging with the `scrollID` no other parameters can be supplied to modify the search request or result.

    Items

    The 'items' block contain a list of search results. Through the itemFormat you can request a number formats to how the result item must be presented. Not all of these itemFormat formats are available in all output formats.

    outputFormat itemFormats mime-type
    protobuf fragmentGraph, summary, grouped application/octet-stream
    json fragmentGraph, summary, json-ld, grouped application/json
    geojson geojson application/geo+json
    The item formats summary, grouped, json-ld, and geojson are currently under development.

    HTTP Request

    GET http://example.com/api/search/v2

    Query Parameters

    Parameter Type Default Description
    q string free text search in the fragments.object field
    rows integer 16 The number of items returned in the response
    scrollID string Key pagination key. After the first response you can use the scrollID to retrieve all subsequent pages
    format string json The output or serialisation format to render the results in. The supported options are: 'json' and 'protobuf'. In the future, 'geobuf' will be added for rendering geo results.
    itemFormat string fragmentGraph How the items should be rendered. The default is fragmentGraph. In the future, the following itemFormats will be added: summary (for a RDF agnostic summary view), JSON-LD (the JSON-based RDF serialisation), grouped (inline the object links in three levels).
    echo string You can echo a number of hidden elements for debugging:
    * es (the elasticsearch query),
    * searchResponse (the elastic search response),
    * searchRequest (protobuf searchRequest encode as JSON),
    * scrollID (decode the searchRequest in the scrollID).
    At this time, the filter and facet options have not been integrated in the default endpoint.

    Get a Specific Record

    You can retrieve a specific record by retrieving the hubID value from search response. 

    curl "http://example.com/api/search/v2/{hubID}"
  -H "Authorization: {apiKEY}"

    This endpoint retrieves a specific RDF Record.

    HTTP Request

    GET http://example.com/api/search/v2/<ID>

    URL Parameters

    Parameter Type Default Description
    ID string None The ID of the RDF Graph to retrieve
    format string json The output or serialisation format to render the results in. The supported options are: 'json' and 'protobuf'. In the future, 'geobuf' will be added for rendering geo results.
    itemFormat string fragmentGraph How the items should be rendered. The default is 'fragmentGraph'. In the future, the following 'itemFormats' will be added: summary (for a RDF agnostic summary view), JSON-LD (the JSON-based RDF serialisation).

    Errors

    The Hub3 V2 Search API uses the following error codes:

    Error Code Meaning
    400 Bad Request -- Your request is invalid.
    401 Unauthorized -- Your API key is wrong.
    403 Forbidden -- The search requested is hidden for administrators only.
    404 Not Found -- The specified record could not be found.
    405 Method Not Allowed -- You tried to access a record with an invalid method.
    406 Not Acceptable -- You requested a format that isn't json.
    410 Gone -- The record requested has been removed from our servers.
    418 I'm a teapot.
    429 Too Many Requests -- You're requesting too many records! Slow down!
    500 Internal Server Error -- We had a problem with our server. Try again later.
    503 Service Unavailable -- We're temporarily offline for maintenance. Please try again later.
